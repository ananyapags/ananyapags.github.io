Making_Models_Actually_Robust

Improving robustness with last-layer retraining while speeding up iteration under tight training time constraints.

Runtime: 80h 00m

PyTorchDDPNCCLSLURMHPC

In a previous post, I mentioned a setup where we had about 10 weeks to improve an algorithm, but each full training run was taking nearly 3 weeks. Here’s the breakdown of that project, where my team and I were tasks in improving the SELF algorithm this paper: Towards Last-layer Retraining for Group Robustness with Fewer Annotations

The Core Problem

A big issue with classification models is that they learn the wrong signals. Instead of focusing on the object itself, they pick up on spurious correlations. For example, a cow vs camel model might rely on grass vs sand instead of the animal.

Where the Problem Actually Lives

What’s interesting is that the problem isn’t usually the entire model. Deep layers still learn useful representations (textures, shapes, structure). The issue often sits in the last layer, where those features get mapped to predictions. That’s where shortcuts get reinforced. This leads to a key idea: instead of retraining everything, focus on fixing the last layer. Baselines like class balancing help, but they don’t directly address why mistakes happen. Group balancing is stronger but requires annotations. Methods like “train twice” try to reweight mistakes, but they’re still fairly coarse. SELF (Selective Last-layer Fine-Tuning) improves on this by identifying which samples matter most. It selects examples based on signals like misclassification or disagreement and retrains only on those. The goal is to focus learning on the parts of the data where the model is uncertain or wrong, rather than treating all samples equally.

Our Improvement

We tested this setup across four text datasets and four image datasets, using BERT for text and ResNet-50 for images. Instead of just using standard accuracy, we focused on worst-group accuracy. This measures how the model performs on its weakest subgroup, which is a better signal of robustness. A model that does well on average but fails on a minority group is still a problem. The original SELF algorithm from the paper has a key limitation. Its hard selection strategy can be noisy. If you select based on misclassification, it can over-focus on outliers. If you select based on disagreement, it can miss clear but important mistakes.

To fix this, we used a hybrid soft-weighted SELF approach.

Instead of selecting a subset of samples, we assign a continuous importance weight to every example. Each sample gets two signals: Loss which captures how wrong the model is Disagreement which captures how uncertain or unstable the prediction is. We combine these using weighting parameters (α for disagreement, β for loss) to create a hybrid importance score. This score is then used in a weighted loss function when retraining the final layer. The key idea is that we do not throw away data. We keep the full dataset but shift focus toward more informative samples. This avoids the noise from hard selection and keeps training stable.

In terms of results, this approach worked well. We achieved around 92–93% overall accuracy, with worst-group accuracy in the mid-80s. More importantly, we reduced the gap between average performance and worst-case performance, which is a better measure of robustness. One key insight was that disagreement (uncertainty) mattered more than raw loss. When we emphasized disagreement (α = 0.8), we got the best robustness. When we emphasized loss too much (β = 0.8), worst-group performance actually dropped. So overall, focusing on where the model is unsure turned out to be more useful than just focusing on where it is wrong.

Back to the Training Problem

The biggest issue early on was iteration speed. Full runs took weeks, so we couldn’t try many ideas. A smaller setup (ResNet) helped since it ran in a few hours, but we still needed to scale to BERT. We also had a SLURM constraint. Jobs would get cut off after ~48 hours, so we added checkpointing and split training into multiple runs. This made things a bit more complex, but much more reliable. Our first big improvement was moving from a single GPU to DDP. With 3 GPUs, training time dropped from ~3 weeks to ~5 days. We also used gradient bucketing in NCCL, which overlaps communication with compute and reduces idle time. But then a new issue showed up. GPU utilization wasn’t steady, it kept dropping between steps. That meant the bottleneck had shifted. The GPUs were no longer the problem, they were waiting on data. The real issue was the data pipeline, specifically loading, preprocessing, and CPU to GPU transfer.
